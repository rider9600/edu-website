# Probability & Statistics Website Roadmap

# 1. Current Product State

The repo already has:

* a Next.js app
* homepage and section pages
* topic MDX content under `content/`
* multiple interactive visualizations
* smoke and Playwright route checks

The next roadmap should build from this state, not from project setup.

---

# 2. Product Goal

Build a strong learning site for:

* probability
* statistics
* interactive intuition

The site should be accurate, visually clear, and easier to navigate than raw notes.

---

# 3. Near-Term Technical Cleanup

Priority items:

* centralize MDX rendering instead of repeating route-level mappings
* split `src/lib/constants.ts` into smaller domain-focused files
* remove encoding issues in text content and UI strings
* reduce duplicate route logic between probability and statistics
* improve the root `README.md`

---

# 4. Content Expansion Priorities

Probability:

* sample space and events
* conditional probability
* Bayes theorem
* random variables
* distributions

Statistics:

* descriptive statistics
* sampling distributions
* confidence intervals
* hypothesis testing
* regression basics

---

# 5. Visualization Priorities

Keep extending the existing visualization library with:

* stronger parameter controls
* better explanation text around charts
* clearer reusable graph wrappers

Candidate next additions:

* sampling distribution visualizer
* confidence interval visualizer
* regression residual explorer

---

# 6. UX Priorities

* make topic navigation consistent across subject pages
* unify breadcrumbs and back navigation
* improve internal topic cross-linking

---

# 7. Documentation Priorities

* keep `docs/` limited to current operational docs
* avoid historical milestone files
* keep structure docs aligned with the real tree
* document MDX component registration more clearly

---

# 8. Release Checklist

Before calling the site stable:

1. type-check passes
2. build passes
3. smoke tests pass
4. key routes have Playwright coverage
5. docs reflect the real structure

---

# 9. Quality Rules

Do:

* prefer clear explanations over breadth
* keep interactive pieces grounded in the lesson
* write docs from the current code, not the planned future code

Do not:

* accumulate duplicated documentation
* keep stale planning notes in operational docs
* add new content categories before the current ones are coherent
